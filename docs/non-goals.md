# DriftKB Non-Goals

This document defines what DriftKB intentionally does not do. These boundaries protect the project from becoming too broad, too private-tool-specific, or unsafe for open source adoption.

## No Remote Service

DriftKB does not provide a hosted service, SaaS control plane, remote validation backend, database, or background worker.

The default operating model is a local Python CLI that can also run in CI.

## No AI Auto-Modification Of Curated KB

DriftKB does not automatically modify curated KB pages with AI-generated content.

Generated stubs or suggestions may be produced by explicit commands, but promotion into curated KB must require human review.

## No Binding To Claude Code

DriftKB does not require Claude Code or any editor agent. The CLI must be usable from a normal terminal and CI environment.

## No Binding To MCP

DriftKB core does not depend on MCP. It may read static cache files generated by external tools, but MCP must remain optional and outside the core validation path.

## No Binding To Java Or Python

DriftKB is not a Java-only or Python-only tool. Built-in Java and Python
adapters are lightweight examples; the core must support configurable adapters
and must not hard-code language assumptions.

## No Binding To Private Directory Structures

DriftKB must not require private or organization-specific directory layouts.

The default KB root is:

```text
docs/kb/
```

Curated and generated KB files use `docs/kb/curated/` and
`docs/kb/generated/` by default.

The default configuration directory is:

```text
.driftkb/
```

Both defaults must be configurable.

## No Default Hook Gap Detection

Gap detection is not part of the default pre-push or pre-commit gate.

Repository hooks should call:

```text
driftkb validate
```

Gap detection should remain an explicit manual workflow, such as:

```text
driftkb gaps detect
```

## No Guarantee Of Full Business Semantics

Verify blocks provide mechanical validation only. They can confirm that files, symbols, routes, commands, or textual patterns still satisfy configured assertions.

They do not guarantee that the complete business meaning of a KB page is correct.
